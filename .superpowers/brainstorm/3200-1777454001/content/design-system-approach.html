Apply “Monochrome + Ember” across the app

Pick how aggressive you want the refactor to be. This choice determines how fast we can make everything consistent.

A

Token-only (fastest)

Update global CSS variables + Tailwind theme mappings. Minimal component edits. UI becomes mostly correct, but a few components may still have “old” radii/shadows.

Pros

  • Lowest risk
  • Quick global consistency
  • Easy to maintain

Cons

  • Some components keep old shapes (e.g. rounded-xl/2xl)
  • Some focus rings may not match “ember glow” yet
B

Token + key components (recommended)

Update tokens, then refactor core primitives (Button, Input, Card, Tabs, Modal, Toast) to match the system rules. Rest inherits automatically.

Pros

  • Consistent look across the whole product
  • Matches the “ember only on interaction” rule
  • Still contained scope (UI primitives)

Cons

  • More code changes than A
  • Requires checking several pages for regressions
C

Full UI sweep (strict)

Update tokens + refactor primitives + touch all pages to enforce typography, spacing, motion budgets, and remove any “decorative” styling.

Pros

  • Most faithful to the design doc
  • Highest visual consistency

Cons

  • Largest scope and highest regression risk
  • Requires more QA time
